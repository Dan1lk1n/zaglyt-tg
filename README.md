# zaglyt-tg

Генеративный Telegram-бот на Go. Он собирает сообщения из групповых чатов,
обучается на них и на их основе генерирует собственные реплики, имитируя
общение. Отвечая, бот старается сохранять контекст того сообщения, на которое
реагирует.

---

## Как это работает — конвейер генерации

Ядро генерации — пакет [modules/z3abp/](modules/z3abp/), функция
`GenerateBestResponse` ([modules/z3abp/z3abp.go](modules/z3abp/z3abp.go)). Это не
простая цепь Маркова по словам, а **гибрид: контекстный поиск + морфологическая
цепь Маркова**. Ответ формируется так:

1. **Сбор данных.** Каждое сообщение в группе сохраняется в таблицу `messages`
   ([handlers/message.go](handlers/message.go), `AppendMessage`). Это и есть корпус,
   на котором бот учится в рамках конкретного чата.
2. **Контекстный поиск.** Входящее сообщение токенизируется и приводится к основам
   слов русским стеммером Snowball. Из истории чата отбираются строки, у которых
   есть хотя бы одна общая основа с запросом (`FilterLines`).
3. **Ранжирование BM25.** Отобранные строки ранжируются алгоритмом BM25 Okapi
   (`RankDocuments`, `k1=1.5`, `b=0.75`) по релевантности запросу. Берутся **top-5**.
4. **Морфологическая цепь Маркова.** Эти 5 строк прогоняются через морфологический
   анализатор Yandex **mystem**, который для каждого слова даёт лемму и граммати­ческие
   признаки. Состояния цепи — это морфо-ключ (лемма + грамматика), а на выход идут
   реальные словоформы (`buildMorphMarkovChain`). Благодаря этому цепь обобщает
   разные формы одного слова, но выдаёт настоящие слова. Строятся переходы порядка 2
   (по паре предыдущих состояний) с откатом на порядок 1.
5. **Генерация.** Случайное блуждание по цепи (`generateMorphMarkovText`): длина
   ответа 2–10 слов, старт с биграммы в 80% случаев (иначе с униграммы), в 15%
   случаев — джиттер-откат на порядок 1 для разнообразия, до 50 попыток получить
   валидную фразу. В конце убираются хвостовая пунктуация и стоп-слова.
6. **Фолбэк.** Если сгенерировать связную фразу не удалось, возвращаются первые
   несколько слов самого релевантного (top-1) сообщения.

**Кэш морфологии.** Весь отобранный корпус заново прогоняется через mystem на
каждый ответ, а вызовы mystem сериализованы через один процесс (единый мьютекс).
Чтобы это не тормозило, результаты анализа кэшируются в LRU-кэше
([modules/z3abp/cache.go](modules/z3abp/cache.go)), ограниченном по числу записей.

**Внешняя зависимость.** Нужен бинарник `mystem` (требует glibc). В Docker он
автоматически скачивается на этапе сборки образа; при локальном запуске должен
лежать рядом с ботом (`./mystem`) или быть доступен в `PATH`.

## Когда бот отвечает (частота общения)

Решение принимает `ShouldRespond`
([modules/helpers/should_respond.go](modules/helpers/should_respond.go)). Бот
отвечает, если выполнено **любое** из условий:

- сообщение является ответом (reply) на сообщение бота;
- сообщение начинается с имени бота (`BOT_NAME`);
- сообщение содержит @username бота;
- **случайно, с настраиваемой вероятностью** (по умолчанию 10%) на любое другое
  сообщение. Вероятность задаётся per-chat командой `/setprob` (см. ниже).

Бот работает **только в группах** — личные чаты и каналы игнорируются (не
сохраняются и не получают ответов). Перед отправкой ответа имитируется набор
текста: показывается статус «печатает…», задержка ≈ `длина ответа × 15 мс`,
ограничена диапазоном 0.5–5 с. Ответ отправляется как reply на сообщение-триггер.

Важно: если бот выключен в чате (`enabled = false`), он **не сохраняет и не
отвечает** — то есть перестаёт и учиться, и писать.

## Команды и функции

| Команда | Область | Кто | Что делает |
| --- | --- | --- | --- |
| (любой текст) | группы | все | Бот учится на сообщении и может ответить (см. раздел выше) |
| `/help` | любая | все | Список команд (команды разработчика видны только разработчику) |
| `/anecdote` | группы | все | Анекдот: случайный шаблон + фраза, сгенерированная из истории чата |
| `/switcher` | группы | админ чата | Инлайн-кнопки включения/выключения бота в чате (поле `enabled`) |
| `/settings` | группы | админ чата | Показывает текущие настройки генерации чата |
| `/setwords <мин> <макс>` | группы | админ чата | Длина ответа в словах (1 ≤ мин ≤ макс ≤ 50) |
| `/setprob <процент>` | группы | админ чата | Шанс случайного ответа, 0–100% |
| `/clear` | группы | админ чата | С подтверждением стирает выученную историю чата (безвозвратно) |
| `/download` | группы | админ чата | Выгружает историю чата файлом `history_<id>.txt` |
| `/whoami` | любая | разработчик | Отвечает «Разработчик.» |
| `/stats` | любая | разработчик | Кол-во чатов в БД: всего / включённых / выключенных |
| `/broadcast <текст>` | любая | разработчик | Рассылка текста по всем активным чатам (в фоне, троттлинг 35 мс) |

**Два уровня прав:**

- **Разработчик** — глобальный владелец бота. Определяется списком ID в env
  `DEVELOPERS` ([modules/helpers/is_user_developer.go](modules/helpers/is_user_developer.go)).
  Управляет `/whoami`, `/stats`, `/broadcast`.
- **Админ чата** — администратор или создатель конкретного Telegram-чата,
  проверяется на лету через `GetChatMember`
  ([modules/helpers/is_user_admin.go](modules/helpers/is_user_admin.go)). Управляет
  `/switcher`, `/settings`, `/setwords`, `/setprob`, `/clear`, `/download`.

Особенности: команды `/start` нет (есть `/help`). Анекдот не использует внешний API —
берётся один из 6 захардкоженных шаблонов ([constants/anecdotes.go](constants/anecdotes.go)),
в который подставляется случайная фраза, сгенерированная по всей истории чата
(`GenerateRandomMarkov` — цепь Маркова без BM25-фильтрации).

## Настройка (переменные окружения)

Конфигурация читается один раз при старте ([configs/config.go](configs/config.go)),
`.env` подхватывается автоматически (см. [.example-env](.example-env)).

| Переменная | Назначение | По умолчанию |
| --- | --- | --- |
| `BOT_TOKEN` | Токен Telegram-бота | **обязательна** |
| `BOT_NAME` | Имя бота (для срабатывания по обращению) | — |
| `DEVELOPERS` | ID разработчиков через пробел | пусто |
| `BOT_MODE` | Режим обновлений: `polling` или `webhook` | `polling` |
| `WEBHOOK_URL` | Публичный HTTPS-URL, куда Telegram шлёт апдейты | обяз. для webhook |
| `WEBHOOK_SECRET` | Секрет, проверяемый в заголовке каждого запроса | обяз. для webhook |
| `WEBHOOK_LISTEN_ADDR` | Локальный адрес HTTP-сервера | `:8080` |
| `WEBHOOK_PATH` | Путь, на котором обслуживается вебхук | `/webhook` |
| `WEBHOOK_IP_ALLOWLIST_ENABLED` | Ограничить входящие IP-диапазонами Telegram | `true` |
| `LOG_LEVEL` | Уровень логов: debug/info/warn/error | `info` |
| `LOG_FORMAT` | Формат логов: `text` или `json` | `text` |
| `MYSTEM_CACHE_SIZE` | Размер LRU-кэша морфологии (0 — выключить) | `20000` |
| `DATABASE_URL` | Полная строка подключения к PostgreSQL (имеет приоритет) | — |
| `DB_DRIVER` | SQL-драйвер | `postgres` |
| `DB_HOST` / `DB_PORT` / `DB_USER` / `DB_PASSWORD` / `DB_NAME` | Параметры БД (если `DATABASE_URL` пуст) | — |

Если `DATABASE_URL` не задана, обязательны `DB_HOST`, `DB_PORT`, `DB_USER`,
`DB_NAME` — иначе бот не стартует.

**Настройки генерации задаются не через env, а per-chat в БД** и меняются прямо из
Telegram: длина ответа (`/setwords`) и вероятность случайного ответа (`/setprob`).
Значения по умолчанию для новых чатов: 2–10 слов, 10%.

## Запуск

### Docker Compose (рекомендуется)

```bash
docker compose up -d
```

Поднимает два сервиса: `bot` и `db` (`postgres:16-alpine`). `mystem` скачивается
и встраивается в образ на этапе сборки, миграции БД применяются автоматически при
старте. Значения задаются через `.env` (`BOT_TOKEN`, `BOT_NAME`, `DEVELOPERS`,
`BOT_MODE`, при необходимости `WEBHOOK_*`). См.
[docker-compose.yml](docker-compose.yml) и [Dockerfile](Dockerfile).

### Локально

1. Установите Go 1.26+ и доступный PostgreSQL.
2. Положите бинарник `mystem` рядом с ботом или в `PATH`.
3. Склонируйте репозиторий:
   ```bash
   git clone https://github.com/pip64/zaglyt-tg
   ```
4. Создайте `.env` на основе [.example-env](.example-env).
5. Запустите:
   ```bash
   go run .
   ```

Таблицы создавать вручную **не нужно** — миграции применяются автоматически при
старте ([repository/migrate.go](repository/migrate.go)).

### Режимы обновлений

- **polling** (по умолчанию) — бот сам опрашивает Telegram, HTTP-сервер не нужен.
- **webhook** — Telegram шлёт апдейты на публичный HTTPS-URL. Нужны `WEBHOOK_URL`
  и `WEBHOOK_SECRET`; поднимается HTTP-сервер с healthcheck `GET /healthz`.

## Миграции БД

Схема БД управляется мигратором [goose](https://github.com/pressly/goose)
(`github.com/pressly/goose/v3`). Раньше здесь был самописный мигратор — теперь
вся логика в [repository/migrate.go](repository/migrate.go) сведена к тонкой
обёртке над goose.

**Как это работает:**

- SQL-файлы лежат в [repository/migrations/](repository/migrations/) и **вшиваются
  в бинарник** через `//go:embed`. Отдельно раскладывать их при деплое не нужно.
- Порядок применения — по имени файла (префикс-версия `0001_`, `0002_`, …).
- Применённые версии goose хранит в служебной таблице `goose_db_version`.
- `Migrate` вызывается из [main.go](main.go) при старте, поэтому **непринятые
  миграции накатываются автоматически** при каждом запуске. Создавать таблицы
  вручную не нужно.

**Формат файла миграции.** Каждый файл содержит секции goose:

```sql
-- +goose Up
CREATE TABLE IF NOT EXISTS example ( ... );

-- +goose Down
DROP TABLE IF EXISTS example;
```

Секция `Up` накатывается, `Down` — откатывает. Для многострочных
операторов (функции, DO-блоки) оборачивайте их в
`-- +goose StatementBegin` / `-- +goose StatementEnd`.

**Добавить новую миграцию.** Создайте следующий по номеру файл в
[repository/migrations/](repository/migrations/), например `0003_add_users.sql`,
с секциями `Up`/`Down`. При следующем старте бота он применится сам. Если
установлен [CLI goose](https://github.com/pressly/goose#install), удобно
сгенерировать заготовку:

```bash
goose -dir repository/migrations -s create add_users sql
```

**Ручное управление (CLI, опционально).** Файлы в стандартном формате goose, так
что тем же корпусом миграций можно управлять из командной строки — например для
отката:

```bash
export DSN="postgres://user:pass@localhost:5432/dbname?sslmode=disable"

goose -dir repository/migrations postgres "$DSN" status   # что применено
goose -dir repository/migrations postgres "$DSN" up       # накатить всё
goose -dir repository/migrations postgres "$DSN" down     # откатить на шаг назад
```

Автоматический старт и CLI работают с одними и теми же файлами, так что
расхождений между «встроенными» и «ручными» миграциями не возникает.

## Архитектура и хранение

Слои: `main` → `handlers` → `app` (фасад над репозиториями) → `repository`
(sqlx + PostgreSQL). Обновления принимаются в режиме polling или webhook; в режиме
webhook работает HTTP-сервер [server/webhook.go](server/webhook.go) с набором
middleware (только POST, лимит тела 1 MiB, allowlist IP Telegram, проверка секрета).

Таблицы в БД:

- `channels` — `channel_id`, `enabled`, `mode`, `min_gen_words`, `max_gen_words`,
  `reply_probability` (настройки чата);
- `messages` — `channel_id`, `text`, `created_at` (выученный корпус сообщений);
- `goose_db_version` — служебная таблица версий миграций (ведётся goose).

## Параметры и точки развития

Настраиваемые per-chat (из Telegram, хранятся в таблице `channels`):

| Параметр | Значение по умолчанию | Команда / поле |
| --- | --- | --- |
| Длина ответа | 2–10 слов | `/setwords` → `min_gen_words`/`max_gen_words` |
| Шанс случайного ответа | 10% | `/setprob` → `reply_probability` |

Пока **захардкожены** в коде (кандидаты на вынос в настройки при развитии):

| Параметр | Значение | Где |
| --- | --- | --- |
| Top-K документов для генерации | 5 | [z3abp.go](modules/z3abp/z3abp.go) (`DefaultConfig`) |
| BM25 `k1` / `b` | 1.5 / 0.75 | [z3abp.go](modules/z3abp/z3abp.go) (`DefaultConfig`) |
| Старт с биграммы / джиттер отката | 80% / 15% | [z3abp.go](modules/z3abp/z3abp.go) |
| Попыток генерации | 50 | [z3abp.go](modules/z3abp/z3abp.go) |
| Верхний предел длины ответа | 50 слов | [handlers/settings.go](handlers/settings.go) (`maxGenWordsCap`) |
| Размер кэша морфологии | 20000 (env `MYSTEM_CACHE_SIZE`) | [z3abp.go](modules/z3abp/z3abp.go) |
| Задержка «печатает…» | длина × 15 мс, 0.5–5 с | [message.go](handlers/message.go) |
| Троттлинг рассылки | 50 мс | [admin.go](handlers/admin.go) |

Возможные направления развития:

- Поле `channels.mode` **зарезервировано в схеме, но нигде не используется** —
  готовая точка для режимов поведения на уровне чата.
- Нет команды `/start`, нет настроек тона и фильтрации контента.
- Backfill истории (обучение на старых сообщениях) **невозможен** через Telegram
  Bot API — бот получает только новые сообщения после добавления в чат.
- Хранится только текст: стикеры, изображения, голосовые и подписи к медиа не
  сохраняются и в обучении не участвуют.

Замеченное несоответствие (без правок в рамках этой задачи): проверка админа в
[modules/helpers/is_user_admin.go](modules/helpers/is_user_admin.go) читает статус
через `member.Owner.Status` независимо от типа участника — потенциально хрупко для
не-владельцев, стоит перепроверить при доработке системы прав.
